01 · Tổng Quan

Bản sao số của hệ thống thực — cập nhật liên tục và có thể ra lệnh điều khiển ngược lại thực tế.

03 · Bản Đồ
Bản Đồ 2D Ngã Tư — Real-time
Canvas lớn cột trái — vẽ lại liên tục bằng requestAnimationFrame. Xe ô tô animated, đèn đổi màu theo chu kỳ thật.
Bạn đang thấy gì? Ngã tư nhìn từ trên xuống. 6 nhánh đường tỏa ra. Màu mỗi nhánh = mật độ xe thực tế. Hình chữ nhật nhỏ di chuyển = xe được vẽ bằng Canvas 2D API, tốc độ giảm khi đường đông.

Code: tốc độ xe tỉ lệ nghịch với mật độ
class Vehicle — update(density)
update(density) {
  // density=0.1 → speedFactor=0.92 (gần tốc độ tối đa)
  // density=0.9 → speedFactor=0.1  (bò như rùa)
  const speedFactor = Math.max(0.1, 1 - density * 0.8);
  this.t += this.speed * speedFactor;
  if (this.t >= 1) this.done = true; // xe ra khỏi màn hình
}
Thử ngay: Kéo slider "Giờ mô phỏng" về 17–19h → xe chạy chậm, màu đỏ. Kéo về 3h sáng → xe chạy nhanh, màu xanh.

06 · State Vector
State Vector x(t) — "Bộ Nhớ" Digital Twin
Tab "State Vector" trong panel mật độ — toàn bộ trạng thái hệ thống dưới dạng JSON có cấu trúc.
x(t) là gì? Snapshot đầy đủ tại thời điểm t: mật độ từng hướng, thời gian đèn, pha hiện tại, sự cố... Từ x(t) AI có thể tính x(t+1) — dự đoán tương lai và chọn hành động tốt nhất.
Ví dụ State Vector lúc 17:30 cao điểm chiều
x(t) — snapshot tại t=17:30:45
x(t) = {
  NS_density: 0.5820  // 58% — Cam 🟠
  EW_density: 0.3100  // 31% — Vàng 🟡
  NE_density: 0.3800  // 38% — Vàng 🟡
  SW_density: 0.2700  // 27% — Xanh 🟢

  NS_wait_sec: 32s   // thời gian xe đã chờ TB
  EW_wait_sec: 17s

  signal: {
    NS_green: 55s  EW_green: 35s
    phase: "NS_GREEN"  tick: 23/90
  }
  incident: null  sim_hour: 17
}
AI đọc x(t) → thử 7 kịch bản → tính x(t+1) cho từng cái → chọn x(t+1) tốt nhất → áp dụng → quan sát x(t+2).
07 · AI Engine
AI What-If Analysis — Layer 4
Panel "Kịch Bản Tối Ưu" — AI thử 7 cách phân bổ thời gian đèn, tính điểm J, chọn J thấp nhất mỗi 3 giây.
What-If là gì? AI tự hỏi "Nếu NS=60s thì mật độ sau 1 chu kỳ sẽ thế nào? Nếu NS=30s thì sao?" — thử 7 kịch bản, chọn tốt nhất.
Hàm mục tiêu J — càng nhỏ càng tốt
scoreState() — tính điểm J
// J = tổng bình phương mật độ — bình phương vì
// 80% tệ hơn nhiều lần so với 2x(40%)
function scoreState(st) {
  return Object.values(st).reduce((a, v) => a + v*v, 0);
}
// NS=0.58, EW=0.31, NE=0.38, SW=0.27
// J = 0.336+0.096+0.144+0.073 = 0.649

08 · REST API
REST API Activity
Panel cột phải — trong demo là giả lập (không gọi server thật). Đây là cách frontend nói chuyện với backend.
REST API là gì? Giao tiếp qua HTTP. GET lấy dữ liệu, POST gửi mới, PUT cập nhật/ra lệnh. Số 200/201/204 là HTTP status code (thành công).
Giải thích từng dòng log API
GET/api/v1/intersection/hk01/state20023ms
→ Đọc trạng thái mật độ hiện tại của ngã tư hk01
POST/api/v1/sensor/batch20138ms
→ Camera gửi batch dữ liệu 4 hướng lên server, tạo mới thành công
GET/api/v1/ai/scenarios?n=720046ms
→ Lấy kết quả 7 kịch bản AI vừa tính xong
PUT/api/v1/signal/hk01/command20412ms
→ Gửi lệnh đổi đèn — 204 = OK không trả dữ liệu về
Code thật — Node.js Express backend
server.js
app.put('/api/v1/signal/:id/command', async (req, res) => {
  const { ns_green, ew_green } = req.body;
  // Gửi lệnh qua MQTT đến bộ điều khiển đèn thật
  mqtt.publish(`traffic/hn/hk01/signal/command`,
    JSON.stringify({ ns_green, ew_green, src: 'AI' })
  );
  res.status(204).send(); // OK, không trả dữ liệu
});
09 · MQTT
MQTT Broker Messages
Panel cột phải — giao thức nhắn tin IoT dùng Publish/Subscribe thay vì Request/Response như HTTP.
MQTT là gì? Giống loa phát thanh: Camera "phát" (publish) lên "kênh" (topic) → Broker giữ và chuyển tiếp → Ai đang "nghe" (subscribe) nhận ngay. Nhẹ hơn HTTP 10× — lý tưởng cho IoT cảm biến nhỏ, mạng kém.
Giải thích từng dòng MQTT log
00:13:47PUBtraffic/hn/hk01/sensor/NS{d:0.58,cnt:142}
→ Camera NS phát mật độ 58% (142 xe) lên broker mỗi giây
00:13:47PUBtraffic/hn/hk01/sensor/EW{d:0.31,cnt:89}
→ Camera EW phát mật độ 31% lên broker
00:13:46SUBtraffic/hn/hk01/signal/cmd{ns_g:55,ew_g:35}
→ Bộ điều khiển đèn NHẬN lệnh từ AI: NS=55s, EW=35s
Code thật — Python camera publisher
camera_publisher.py
import paho.mqtt.client as mqtt
client = mqtt.Client()
client.connect("broker.twin.local", 1883)
while True:
    density = count_vehicles() / max_capacity
    client.publish("traffic/hn/hk01/sensor/NS",
        f'{{"d":{density:.3f}}}', qos=0)
    time.sleep(1)  # gửi mỗi 1 giây
10 · InfluxDB
InfluxDB — TimeSeries Database
Panel cột phải — database chuyên lưu dữ liệu theo thời gian. Mỗi chu kỳ ghi ~4 dòng vào DB.
Tại sao không dùng MySQL? 4 cảm biến × 1 lần/giây × 30 ngày = 10 triệu rows. MySQL sẽ chậm dần. InfluxDB ghi hàng triệu điểm/giây, tự nén data, query theo thời gian trong <100ms.
Giải thích dòng InfluxDB Line Protocol
Format: measurement,tags fields timestamp
traffic_density,loc=hk01,dir=NS value=0.3841782839646
→ Lưu mật độ NS=38.4% tại ngã tư hk01 vào lúc timestamp đó
signal_state,loc=hk01 ns_g=55,ew_g=351782839645
throughput,loc=hk01 veh_hr=1391782839644
Query Flux — lấy mật độ 1 giờ qua để vẽ chart
Flux Query Language
from(bucket: "traffic_hn")
  |> range(start: -1h)
  |> filter(fn: (r) => r.dir == "NS")
  |> aggregateWindow(every: 1m, fn: mean)
// Kết quả: [{time:"17:00",value:0.32},{time:"17:01",...}]
// Dùng để vẽ biểu đồ lịch sử mật độ 30 chu kỳ

12 · Kết Luận
Feedback Loop — Vòng Điều Khiển Khép Kín
Khái niệm trung tâm của hệ thống — không chỉ giám sát mà còn tự động tác động lại thực tế.
Điểm mấu chốt: Giám sát thông thường chỉ đọc. Digital Twin còn ghi ngược lại — AI quyết định → ảnh hưởng thực tế → thực tế đổi → dữ liệu mới → AI tính lại. Vòng lặp khép kín liên tục.
Chu trình 1 vòng lặp — mỗi ~3 giây trong demo
① Camera đọc
NS=58%, 142 xe
→ MQTT PUB: sensor/NS {d:0.58}
② Twin Engine
Cập nhật x(t)
→ Lưu InfluxDB, x(t)={NS:0.58, EW:0.31...}
③ AI tính toán
7 kịch bản → J
→ Best: NS=55s/EW=35s, J=0.085 (thấp nhất)
④ Ra lệnh đèn
Feedback → vật lý
→ PUT /api/signal → MQTT → Đèn vật lý đổi timing ngay
↺ Lặp lại từ ① — đèn NS=55s ảnh hưởng đến mật độ chu kỳ tiếp theo
Tổng kết: Đây là closed-loop control system — cùng nguyên lý với Tesla Autopilot và điều hòa thông minh. Digital Twin áp dụng nguyên lý đó cho hạ tầng đô thị. Khi bạn bấm "Áp Dụng Đề Xuất" trong demo, đó chính là khoảnh khắc Feedback Loop đóng lại!
